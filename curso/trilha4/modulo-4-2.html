🎯 95% Confidence - Deep Dive

O 95% Confidence vai alem do basico quando voce o aprofunda. Neste topico, exploramos variacoes avancadas: ajustar o percentual de confianca, direcionar para areas especificas do projeto e combinar com instrucoes de CLAUDE.md para automacao completa.

🎯 Variacao Avancada - Direcionado 

Antes de tocar em qualquer coisa, estude especificamente:
1. A camada de autenticacao (auth/, middleware/, guards/)
2. Como os tokens sao gerados, validados e renovados
3. Os testes existentes para auth
4. As variaveis de ambiente relacionadas

Preciso de 95% de confianca de que voce entende o fluxo completo de auth antes de modificar qualquer coisa. Me mostre um diagrama do fluxo atual.
  • • Direcionar o estudo evita que o Claude perca tempo lendo arquivos irrelevantes
  • • Pedir um diagrama do fluxo forca o Claude a sintetizar o entendimento

💡 Dica Pratica

Para projetos grandes, use 99% Confidence para areas criticas (pagamentos, auth, dados sensiveis) e 90% Confidence para areas menos criticas (UI, formatacao). O percentual comunica a profundidade de estudo esperada.

🚀 Project Kickoff - Projetos Novos

O Project Kickoff e o prompt para quando voce esta comecando um projeto do zero. Ele guia o Claude para criar a estrutura inicial com padroes solidos desde o comeco, evitando divida tecnica precoce.

🎯 Template do Prompt 

Estou comecando um projeto novo. Antes de criar qualquer arquivo:

1. Me faca perguntas sobre: stack, escopo, requisitos, deploy target
2. Baseado nas respostas, proponha a estrutura de pastas
3. Defina os padroes: naming conventions, estrutura de imports, error handling
4. Crie o CLAUDE.md com essas convencoes documentadas
5. So entao comece a implementar, um passo de cada vez

Projeto: [descricao do projeto]
Stack desejada: [tecnologias]
  • • Forca o Claude a perguntar antes de assumir decisoes arquiteturais
  • • Cria o CLAUDE.md desde o inicio - as convencoes ficam documentadas

✓ O que FAZER

  • ✓Responder as perguntas do Claude com detalhes
  • ✓Revisar a estrutura de pastas proposta
  • ✓Validar o CLAUDE.md antes de prosseguir

✗ O que NAO fazer

  • ✗Pular as perguntas e mandar criar tudo de uma vez
  • ✗Deixar o Claude escolher a stack sem sua orientacao
  • ✗Comecar a codar sem definir convencoes primeiro

🗺️ Codebase Onboarding - Projetos Existentes

O Codebase Onboarding e para quando voce entra em um projeto que ja existe. Ele faz o Claude mapear a arquitetura, identificar padroes e criar um mapa mental do projeto para voce.

🎯 Template do Prompt 

Estou entrando neste projeto pela primeira vez. Faca um onboarding completo:

1. Leia package.json / requirements.txt / go.mod (ou equivalente)
2. Mapeie a estrutura de pastas e o proposito de cada diretorio
3. Identifique os padroes de arquitetura (MVC, Clean, Hexagonal, etc)
4. Liste os endpoints/rotas principais
5. Identifique o fluxo de dados (entrada → processamento → saida)
6. Aponte areas de risco ou divida tecnica visivel

Me de um resumo executivo de 1 paragrafo seguido do mapeamento detalhado.

💡 Dica Pratica

Salve o output do Codebase Onboarding no CLAUDE.md do projeto. Assim, sessoes futuras ja comecam com esse mapeamento e nao precisam refazer o estudo. Atualize conforme o projeto evolui.

4

🎬 Estrategia de Primeira Sessao

A primeira sessao define o tom de todo o trabalho com o Claude em um projeto. Se voce comecar mal, toda sessao subsequente herda problemas. Aqui esta a sequencia ideal para a primeirissima interacao com qualquer projeto.

🎯 Sequencia da Primeira Sessao

1
Rode /init - Cria o CLAUDE.md base automaticamente
2
Use Codebase Onboarding - Mapeie a arquitetura completa
3
Enriqueca o CLAUDE.md - Adicione o mapeamento ao arquivo de contexto
4
Faca uma tarefa pequena - Valide que o Claude entende o projeto com uma mudanca simples

💡 Dica Pratica

A tarefa pequena de validacao deve tocar uma area central do projeto. Se o Claude acertar, voce sabe que o entendimento esta correto. Se errar, voce identifica gaps no CLAUDE.md e corrige antes de tarefas maiores.

5

❓ Padroes de Perguntas Estrategicas

Fazer as perguntas certas ao Claude no inicio da sessao economiza horas de retrabalho. Estes sao os padroes de perguntas que extraem o maximo de informacao relevante antes de iniciar qualquer trabalho.

🎯 Perguntas que Voce Deve Fazer 

Perguntas de Arquitetura:
- "Quais sao as dependencias criticas deste modulo?"
- "Se eu mudar X, o que pode quebrar?"
- "Existe algum padrao nao-obvio neste codigo que eu deveria saber?"

Perguntas de Contexto:
- "Como essa feature se conecta com o resto do sistema?"
- "Quais testes cobrem essa area?"
- "Existe documentacao inline que eu deveria ler?"

Perguntas de Risco:
- "Qual o pior cenario se essa mudanca tiver um bug?"
- "Tem alguma race condition ou estado compartilhado aqui?"
- "Essa area tem historico de bugs recentes nos commits?"

💡 Dica Pratica

Adicione suas perguntas favoritas ao CLAUDE.md como checklist automatico. Exemplo: "Antes de qualquer mudanca, sempre responda: quais arquivos serao afetados? Quais testes existem? Qual o risco?"

6

🚫 Evitando Pontos Cegos

O Claude tem pontos cegos previsíveis no inicio de sessoes. Ele tende a assumir padroes comuns, ignorar configuracoes especificas do projeto e nao verificar versoes de dependencias. Saber disso permite que voce previna erros comuns.

🎯 Pontos Cegos Comuns e Como Prevenir

  • • Versoes de dependencias - Instrua: "Verifique a versao no package.json antes de usar qualquer API"
  • • Configuracoes de ambiente - Instrua: "Leia .env.example antes de assumir variaveis de ambiente"
  • • Convencoes do time - Instrua: "Siga o estilo dos arquivos existentes, nao invente padroes novos"
  • • Testes existentes - Instrua: "Rode os testes antes e depois de qualquer mudanca"

💡 Dica Pratica

Todas essas instrucoes de prevencao devem ir para o CLAUDE.md. Assim, o Claude lê automaticamente a cada sessao e voce nao precisa repetir. O CLAUDE.md e sua arma contra os pontos cegos.
